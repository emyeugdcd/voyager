# Voyager 
### Production-Grade Cloud Infrastructure on Azure 

Voyager migrates an application stack (React frontend, Go backend, PostgreSQL) to a secure, private cloud environment on Microsoft Azure. Built as the capstone of an 8-project DevOps module, it represents a complete infrastructure-as-code implementation covering networking, Kubernetes, GitOps delivery, secrets management, observability, and CI/CD.

---

## Project Overview
 
Voyager provisions a **private, multi-environment cloud platform** across isolated Azure resource groups (Shared, Test, Prod). No resource in the cluster has a public IP; all inbound paths go through a controlled entry point, and all outbound paths go through a NAT Gateway. Credentials never appear in code; they are generated, stored in Key Vault, and injected into pods at runtime via the External Secrets Operator.
 
The platform enforces:
- **Least privilege** — every workload has its own Managed Identity with only the permissions its job requires
- **Private-only endpoints** — AKS control plane, nodes, and database are all inaccessible from the public internet
- **GitOps delivery** — the cluster's desired state lives in Git; ArgoCD continuously reconciles reality to match it
- **Automated credential rotation** — database passwords are generated by Terraform, stored in Key Vault, and synced to Kubernetes Secrets by ESO without human handling

## Technology Stack
 
| Layer | Technology | Notes |
|---|---|---|
| Cloud | Microsoft Azure | northeurope region |
| IaC | Terraform ~3.110 | Modular, remote state in Azure Blob |
| Kubernetes | AKS (private) | Azure CNI, Workload Identity, Defender |
| GitOps | ArgoCD | App of Apps pattern |
| Secrets | External Secrets Operator + Key Vault | No credentials in Git or pod specs |
| DNS | External DNS + Azure DNS | Auto-creates DNS records from Ingress |
| Metrics | Prometheus + kube-state-metrics | Pinned to monitoring node pool |
| Logs | Loki + Promtail | Long-term storage in Azure Blob |
| Dashboards | Grafana | Cluster health, app logs, Postgres metrics |
| CI/CD | GitLab CI + Kaniko | Build, push, ArgoCD sync, manual prod gate |
| Database | Azure Database for PostgreSQL Flexible Server | Private endpoint, PITR enabled |
| Registry | Azure Container Registry (ACR) | Daily purge of untagged images |
| Cluster Access | Linux VM Jumphost (B1s) | Replaces Azure Bastion — ~95% cost saving |

│   ├── argocd/              # ArgoCD installation values and App-of-Apps templates

│   │   ├── apps/            # App of Apps root. ArgoCD watches this directory
│   ├── external-dns/        # External DNS Helm configurations
│   ├── external-secrets/    # External Secrets Operator values and secret stores
│   └── monitoring/          # Observability values (Prometheus, Loki, Grafana)
├── scripts/                 # Operational helper scripts
│   └── teardown.sh          # Destroys environments in reverse dependency order
└── terraform/               # Infrastructure as Code
    ├── environments/        # Root environment configurations
    │   ├── shared/          # ACR + remote state storage account
    │   ├── test/            # Isolated test environment (single-node, 7-day retention)
    │   └── prod/            # Production environment (HA nodes, HA database, 30-day retention)
    └── modules/             # Reusable infrastructure code blocks
        ├── networking/      # VNet, Subnet delegation, and NAT Gateway modules
        ├── aks/             # Azure Kubernetes Service module (3 node pools + Workload Identity)
        ├── database/        # PostgreSQL Flexible Server (private, HA, PITR)
        ├── keyvault/        # Key Vault and auto-generated database secrets module
        ├── dns/             # Azure DNS zones (public + private) + VNet links
        └── jumphost/        # Secure VM Jumphost provisioning module
```

## Deployment Guide
 
### 1. Authenticate with Azure
 
```bash
az login
az account set --subscription "<your-subscription-id>"
```
 
### 2. Bootstrap Remote State and Shared Registry
 
The shared environment creates the Terraform state storage account and the Azure Container Registry. It must be applied first — every other environment depends on its outputs.
 
```bash
cd terraform/environments/shared
terraform init
terraform apply
```
 
### 3. Deploy an Environment
 
```bash
cd terraform/environments/test
terraform init
terraform apply -var="ssh_public_key=$(cat ~/.ssh/id_rsa.pub)"
```
 
Repeat with `environments/prod` for the production environment (Note, for studying and demonstration purposes, only test environment is recommended since the free Azure plan only allows 4 VMs). After apply, capture all outputs:
 
```bash
terraform output
```
 
### 4. Inject Workload Identity Client IDs
 
Terraform provisions Managed Identities for ESO, External DNS, and Loki. Their client IDs must be placed into the corresponding Helm values files before ArgoCD syncs:
 
| Output variable | File to update |
|---|---|
| `eso_identity_client_id` | `kubernetes/external-secrets/values-test.yaml` |
| `external_dns_identity_client_id` | `kubernetes/external-dns/values-test.yaml` |
| `loki_identity_client_id` | `kubernetes/monitoring/loki/helm-values.yaml` |
| `loki_storage_account_name` | `kubernetes/monitoring/loki/helm-values.yaml` |
 
### 5. Bootstrap ArgoCD

Install ArgoCD into the cluster (directly from your local terminal for test and in the root directory /voyager):
- **Fetch credentials**:
```bash
az aks get-credentials --resource-group voyager-test-rg --name voyager-test-aks --admin
```

- **Create namespaces**:

```bash
kubectl create namespace argocd
kubectl create namespace external-secrets
kubectl create namespace external-dns
kubectl create namespace monitoring
kubectl create namespace sampleapp
```

- **Install ArgoCD** (using the new values-test.yaml override):

```bash
helm repo add argo https://argoproj.github.io/argo-helm
helm repo update
helm upgrade --install argocd argo/argo-cd -n argocd -f kubernetes/argocd/install/values.yaml -f kubernetes/argocd/install/values-test.yaml

- **Apply the root App of Apps**:

```bash
kubectl apply -f kubernetes/argocd/apps/root-app.yaml -n argocd
```
 
ArgoCD will detect and sync all downstream applications: ESO, External DNS, Prometheus, Loki, Grafana, and the frontend/backend applications.

## Cluster Access

Access differs between environments to accommodate subscription limits:

* **Production Environment**: The AKS control plane and worker nodes are fully private. Access is mediated via the secure VM Jumphost using SSH port-forwarding.
* **Test Environment**: Due to regional VM core limits in trial accounts, the Jumphost is removed and the control plane API server is configured as public. You can configure `kubectl` directly from your local machine, while worker nodes and databases remain fully private and protected within the VNet.

### Accessing the Production Cluster (Jumphost)

#### 1. Start the Jumphost

```bash
# Start the VM (only when you need access)
az vm start \
  --resource-group voyager-prod-rg \
  --name voyager-prod-jumphost

# Confirm it is running and get the public IP
az vm show \
  --resource-group voyager-prod-rg \
  --name voyager-prod-jumphost \
  --show-details \
  --query "[powerState, publicIps]" \
  --output table
```

#### 2. Connect and Configure kubectl

```bash
# SSH with port forwarding for local kubectl access
ssh -i ~/.ssh/id_rsa -L 6443:localhost:6443 azureuser@<prod-jumphost-public-ip>

# Fetch cluster credentials from inside the VM
az aks get-credentials \
  --resource-group voyager-prod-rg \
  --name voyager-prod-aks \
  --admin

kubectl get nodes
```

#### 3. Stop/Deallocate the Jumphost (Important)

Always **deallocate** the VM when finished to avoid compute charges.

```bash
az vm deallocate \
  --resource-group voyager-prod-rg \
  --name voyager-prod-jumphost
```

### Accessing the Test Cluster (Direct Access)

Since the Test control plane is public, you can connect directly from your local machine:

```bash
# Fetch cluster credentials directly on your local terminal
az aks get-credentials \
  --resource-group voyager-test-rg \
  --name voyager-test-aks \
  --admin

kubectl get nodes
```
 
## CI/CD Pipeline
 
The GitLab CI pipeline (`.gitlab-ci.yml`) runs three stages automatically and one with a manual gate:
 
| Stage | Trigger | Action |
|---|---|---|
| **test** | Every commit | Go unit and integration tests |
| **build** | Push to `main` | Build frontend + backend images with Kaniko, push to ACR |
| **sync-test** | After build | ArgoCD CLI triggers sync to test environment |
| **sync-prod** | Manual approval | Deploys to prod only after test sync succeeds |
 
The pipeline uses the Terraform-provisioned CI service principal credentials stored as GitLab CI variables (`ARM_CLIENT_ID`, `ARM_CLIENT_SECRET`, `ARM_TENANT_ID`, `ARM_SUBSCRIPTION_ID`).

## Observability
 
Three Grafana dashboards are provisioned automatically:
 
- **Cluster Health** — node CPU/memory, pod restarts, pending pods, PVC usage
- **Application Logs** — Loki-backed log explorer scoped to frontend and backend namespaces
- **PostgreSQL Metrics** — connection counts, query latency, replication lag, disk usage via postgres-exporter
Prometheus alerting rules fire to a configured webhook (Slack/Discord) on: pod crash-looping, node memory pressure, database disk above 80%, and ArgoCD sync failures.
 
Log retention: 1 month for metrics, 1 year for logs (stored in Azure Blob with lifecycle policy).

## Disaster Recovery
 
Two runbooks are maintained in [`docs/runbooks/`](docs/runbooks/):
 
**Rollback:** Deploy a broken image → ArgoCD detects health check failure → trigger rollback via ArgoCD CLI or UI → previous ReplicaSet restored. Full procedure in [`docs/runbooks/rollback.md`](docs/runbooks/rollback.md).
 
**PITR Recovery:** Simulate database failure → restore PostgreSQL Flexible Server to a point-in-time snapshot → reconnect application via updated Key Vault secret → ESO propagates updated credentials to pods without redeployment. Full procedure in [`docs/runbooks/pitr-recovery.md`](docs/runbooks/pitr-recovery.md).

## Teardown
 
To destroy all resources and stop all billing charges:
 
```bash
./scripts/teardown.sh
```
 
The script destroys environments in reverse dependency order: Prod → Test → Shared. Shared is prompted separately to avoid accidentally destroying the container registry if it holds images you want to keep.

## Design Decisions
 
Notable architecture decisions are documented in [`docs/adr/`](docs/adr/):
 
- **[ADR-001](docs/adr/001-azure-provider.md)** — Why Azure over AWS/GCP (Nordic job market, AKS maturity, Findata/GDPR region availability)
- **Jumphost over Bastion** — Cost: ~€1/month deallocated vs ~€130/month for Azure Bastion Basic. Same security model, manual lifecycle discipline required.
- **Basic ACR SKU** — Sufficient for a capstone; a production deployment would use Standard for content trust and increased storage limits.
- **Azure CNI over kubenet** — Required for private AKS with internal load balancers and direct pod routability within the VNet.
